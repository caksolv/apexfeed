Debugging Salesforce Governor Limit Errors: Practical Fixes and Prevention Tips

Salesforce governor limits are rules that cap resource usage in Apex code to protect the multi-tenant environment. Because many customers share the same server resources, Salesforce enforces limits to prevent any single org’s code from hogging databases, memory, CPU, or network bandwidth.

For example, there’s a limit on how many queries or DML operations you can perform in one transaction. If a governor limit is exceeded, Salesforce throws a runtime exception and halts execution. These limits ensure fairness and stability, but they can frustrate developers when hit unexpectedly.

Common Salesforce Governor Limit Errors

“Too Many SOQL Queries: 101”

Salesforce allows at most 100 SOQL queries per synchronous transaction. This often occurs when a SOQL query is inside a loop. Move queries outside the loop and bulkify operations.

“Too Many DML Statements: 151”

There’s a 150 DML limit per Apex transaction. Avoid doing DML in loops. Instead, collect records in a list and commit them in bulk.

“Apex CPU Time Limit Exceeded”

The CPU time limit is 10,000 ms for synchronous Apex. Optimize logic, minimize recursion, and shift work to async jobs where possible.

“Apex Heap Size Too Large”

Heap size is limited to 6 MB (sync) and 12 MB (async). Query only necessary fields, use SOQL for loops, and batch large operations.

“Too Many Callouts: 101”

Only 100 callouts are allowed per transaction. Avoid callouts inside loops; use batch jobs or consolidate requests.

Debugging Techniques for Governor Limit Errors

  • Check Debug Logs: Logs show resource usage and exceptions before the failure point.
  • Developer Console Analysis: Use Execution Overview and Limits methods to monitor behavior.
  • Checkpoints and Heap Inspector: Helpful for tracking memory use and object states.
  • Isolate in Execute Anonymous: Run minimal examples to reproduce and isolate problems.

Examples of Violations and Fixes

Example 1: SOQL Inside a Loop

Violation:

for (Contact c : contacts) {
  Account acc = [SELECT Name, Industry FROM Account WHERE Id = :c.AccountId];
}

Fix: Move SOQL outside the loop and use a map:

Set<Id> acctIds = new Set<Id>();
for (Contact c : contacts) {
  acctIds.add(c.AccountId);
}

Map<Id, Account> accounts = new Map<Id, Account>(
  [SELECT Name, Industry FROM Account WHERE Id IN :acctIds]
);

for (Contact c : contacts) {
  Account acc = accounts.get(c.AccountId);
}

Example 2: DML Inside a Loop

Violation:

for (Contact c : contacts) {
  Task t = new Task(WhoId = c.Id, Subject = 'Follow Up');
  insert t;
}

Fix: Batch the DML operations:

List<Task> tasks = new List<Task>();
for (Contact c : contacts) {
  tasks.add(new Task(WhoId = c.Id, Subject = 'Follow Up'));
}
insert tasks;
